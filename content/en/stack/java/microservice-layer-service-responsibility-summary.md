# 微服务架构不是简单拆服务

微服务架构的核心不是“把一个系统拆成很多服务”，而是把系统按照业务边界、技术职责、稳定性要求和团队协作方式进行重新组织。

如果只是把单体项目拆成多个仓库、多个进程，但请求链路、数据边界、配置方式、发布流程、故障隔离都没有治理好，那么系统复杂度只会被放大。

我理解的微服务落地，需要回答几个问题：

```shell
请求从哪里进来？
网关负责什么？
业务服务之间如何协作？
哪些数据必须强一致？
哪些链路可以异步化？
配置如何管理？
流量如何治理？
故障如何定位？
发布如何回滚？
```

只有这些问题有明确答案，微服务才算真正进入可维护状态。

---

# 一、整体分层视图

一个比较清晰的微服务系统，可以按下面几层理解：

```shell
用户 / 前端 / 第三方系统
        ↓
入口层：Nginx / SLB / CDN
        ↓
网关层：Spring Cloud Gateway
        ↓
治理层：鉴权 / 限流 / 熔断 / 灰度 / 日志
        ↓
业务服务层：用户、订单、资金、游戏、后台、报表等领域服务
        ↓
异步消息层：RabbitMQ / Kafka / Outbox / 消费者
        ↓
数据存储层：MySQL / Redis / ClickHouse / MinIO
        ↓
可观测层：日志 / 指标 / 链路追踪 / 告警
```

每一层都要有自己的边界，不能所有事情都塞进业务服务，也不能让网关承担过多业务逻辑。

---

# 二、入口层：负责流量进入和基础转发

入口层通常由 Nginx、SLB、CDN 等组成。

它的核心职责是：

```shell
域名解析
HTTPS 证书
静态资源代理
反向代理
负载均衡
前端 history fallback
基础安全控制
```

入口层不要写太多业务判断，它应该保持简单稳定。

例如：

```shell
/api/frontend/**  → gateway
/api/backend/**   → gateway
/api/manage/**    → gateway
/assets/**        → static
```

这样做的好处是：

- 请求入口统一，排障时能先确认流量是否进入系统。
- 前端、后台、总后台的路径边界清晰。
- 后面服务拆分时，不需要频繁调整外部访问方式。

入口层的目标不是“聪明”，而是“稳定、清晰、少变化”。

---

# 三、网关层：统一入口与请求治理中心

网关层通常使用 Spring Cloud Gateway。

网关不是业务服务，它更像系统的第一道业务边界。

核心职责：

```shell
统一路由
JWT 鉴权
用户上下文透传
站点 / 租户解析
限流
灰度路由
统一错误返回
请求日志
链路追踪 TraceId 生成与透传
```

例如网关可以按路径分发：

```shell
/api/frontend/user/**   → user-service
/api/frontend/order/**  → order-service
/api/backend/**         → backend-service
/api/manage/**          → manage-service
```

网关层最重要的原则：

```shell
网关做通用治理，不做复杂业务。
```

适合放在网关的逻辑：

- 登录态校验
- Token 解析
- 请求头补充
- 黑白名单
- IP 限流
- 租户域名解析
- 统一异常包装

不适合放在网关的逻辑：

- 订单状态流转
- 资金扣减
- 复杂权限计算
- 报表计算
- 跨服务业务编排

一旦网关承担太多业务逻辑，它就会变成新的单体，而且所有请求都会被它拖慢。

---

# 四、服务注册与配置层：Nacos 的职责

Nacos 在微服务架构中通常承担两个职责：

```shell
服务注册与发现
配置中心
```

## 1、服务注册与发现

每个微服务启动后，把自己的服务名、IP、端口、健康状态注册到 Nacos。

调用方不再写死 IP，而是通过服务名调用：

```shell
lb://user-service
lb://order-service
lb://balance-service
```

这样服务扩容、重启、迁移时，请求方不用修改配置。

## 2、配置中心

配置中心适合管理：

```shell
数据库连接
Redis 地址
MQ 地址
限流规则
开关配置
第三方接口地址
业务参数
```

配置中心的价值在于：

- 配置集中管理。
- 不同环境隔离。
- 配置变更可以动态刷新。
- 敏感配置可以统一治理。

但配置中心也要注意边界：

```shell
配置可以动态调整，但不能代替发布流程。
```

对于高风险配置，比如资金比例、支付通道、核心限流阈值，要有审批、灰度和回滚记录。

---

# 五、业务服务层：按照领域职责拆分

业务服务层是微服务架构的核心。

拆服务不要按技术模块拆，而要按业务领域拆。

例如：

```shell
user-service       用户、账号、登录态、用户资料
order-service      订单创建、订单状态、订单查询
balance-service    余额、流水、充值、提现、资金变动
game-service       游戏记录、投注、结算、游戏回调
manage-service     总后台管理、运营配置、管理员权限
report-service     报表统计、聚合查询、导出任务
```

每个服务都要能回答：

```shell
我负责什么？
我不负责什么？
我的上游是谁？
我的下游是谁？
我的核心数据是什么？
失败时如何降级？
发布时如何回滚？
```

如果一个服务的职责说不清楚，那么这个服务迟早会变成“大杂烩服务”。

---

# 六、服务职责划分方法

我通常用下面几个标准判断服务边界。

## 1、按业务领域拆

例如用户、订单、资金、游戏、后台管理，本身就是不同领域。

领域边界越清晰，服务边界越稳定。

## 2、按数据归属拆

一个核心原则：

```shell
谁拥有数据，谁负责修改。
```

例如余额数据归 balance-service 管，那么其它服务不能直接改余额表，只能调用 balance-service 的接口。

这样可以避免：

- 多个服务同时修改同一张表。
- 业务规则散落各处。
- 出问题时找不到真正的责任服务。

## 3、按故障影响拆

高风险链路要单独隔离。

例如资金服务、游戏结算、支付回调、消息补偿，都不应该和普通查询接口混在一起。

这样做的好处是：

- 热点服务可以单独扩容。
- 故障服务可以快速隔离。
- 发布风险不会扩散到整个系统。

## 4、按发布频率拆

变化频繁的模块和稳定模块不要绑死在一起。

例如运营后台经常改，资金核心逻辑相对稳定，两者就不适合放在一个服务里。

# 七、数据层：强一致与最终一致要分清楚

微服务最容易出问题的地方是数据一致性。

不是所有场景都需要强一致。

## 1、强一致场景

适合使用数据库事务：

```shell
余额扣减
订单状态变更
支付状态更新
提现审核
核心账务流水
```

这些场景要保证：

- 要么全部成功。
- 要么全部失败。
- 失败要能重试。
- 每次变更要有流水记录。

## 2、最终一致场景

适合使用 MQ、Outbox、定时补偿：

```shell
报表统计
ClickHouse 同步
消息通知
积分更新
运营分析
搜索索引更新
```

最终一致不是“不管一致”，而是通过机制保证最后能对齐。

常见方案：

```shell
本地事务写业务表
        ↓
同事务写 outbox 表
        ↓
relay 任务发送 MQ
        ↓
消费者处理消息
        ↓
失败记录 + 重试 + 补偿
```

Outbox 的关键价值是：

```shell
避免业务数据写成功，但消息没有发出去。
```

# 八、异步消息层：削峰、解耦、补偿

MQ 的作用不是让系统变复杂，而是把不需要同步完成的事情从主链路中拆出去。

适合异步化的场景：

```shell
写报表
同步 ClickHouse
发送通知
生成导出文件
风控记录
操作日志
数据补偿
```

MQ 消费者要重点关注：

- 幂等：同一条消息重复消费不能造成重复扣款或重复写入。
- 重试：临时失败要能自动重试。
- 死信：一直失败的消息要进入死信队列或失败表。
- 监控：积压数量、失败数量、消费耗时都要可见。
- 补偿：消费者挂了之后能继续追数据。

消息系统的底线：

```shell
可以延迟，但不能丢。
可以重试，但不能重复产生副作用。
```

# 九、治理层：限流、熔断、降级

微服务不是服务越多越稳定，服务越多，调用链路越长，故障传播风险也越高。

所以需要治理层。

Sentinel 常用来处理：

```shell
限流
熔断
降级
热点参数保护
系统自适应保护
```

## 1、限流

限制瞬时流量，保护服务不被打垮。

适合入口：

- 登录接口
- 下单接口
- 支付回调
- 查询报表
- 导出任务

## 2、熔断

当下游服务异常比例过高或响应太慢时，调用方快速失败，避免线程堆积。

熔断的意义是：

```shell
不要让一个慢服务拖死整条链路。
```

## 3、降级

降级不是失败，而是保核心链路。

例如：

- 报表失败，不影响用户登录。
- 推荐失败，返回默认列表。
- 非核心统计失败，先记录日志后补偿。
- 外部接口超时，返回处理中状态。

系统稳定性设计的核心思想：

```shell
优先保护核心交易链路。
非核心能力可以延迟、降级、补偿。
```

# 十、可观测层：日志、指标、链路追踪

微服务系统出问题时，最怕的是“不知道请求走到哪里了”。

所以必须建设可观测能力。

## 1、日志

日志要包含：

```shell
trace_id
user_id
tenant_id
service_name
api_path
request_id
cost_ms
error_code
error_message
```

日志不是越多越好，而是关键字段要能串起来。

## 2、指标

重点关注：

```shell
QPS
错误率
P95 / P99 延迟
CPU / 内存
线程池
连接池
MQ 积压
DB 慢查询
```

指标用于发现问题，日志用于定位问题，链路追踪用于还原调用路径。

## 3、链路追踪

TraceId 要从网关开始生成，并一路透传到所有服务。

一条完整链路应该能看到：

```shell
gateway
  → user-service
  → order-service
  → balance-service
  → mq-consumer
  → clickhouse-sync
```

这样出现超时、错误、失败时，才能快速定位到底是哪一层的问题。

# 十一、发布与回滚：架构必须可操作

架构设计不能只停留在图上，必须能发布、能回滚、能排障。

我认为一个服务上线前至少要具备：

```shell
健康检查
配置隔离
日志输出
指标上报
接口超时
错误码规范
回滚方案
数据库变更脚本
灰度验证方式
```

发布策略可以按风险分层：

```shell
低风险：普通配置、静态页面、小功能
中风险：业务接口、查询逻辑、后台功能
高风险：资金、支付、结算、消息一致性、数据库结构
```

高风险发布一定要有：

- 发布前备份。
- 小流量验证。
- 明确回滚命令。
- 关键指标观察。
- 失败补偿方案。

真正成熟的架构，不是永远不出问题，而是：

```shell
出问题能快速发现
发现后能快速定位
定位后能快速止血
止血后能沉淀机制
```

# 十二、架构方法总结

微服务架构落地，我会遵循下面几个方法。

## 1、先收口，再拆分

先把入口收口到网关，再逐步拆服务。

不要一开始就拆很多服务，否则排障、配置、发布都会失控。

## 2、先核心链路，再边缘能力

优先治理：

```shell
登录
鉴权
订单
资金
支付
游戏结算
消息一致性
```

边缘能力可以后面逐步优化。

## 3、先可观测，再优化

没有日志、指标、链路追踪之前，不要盲目优化。

否则优化只是猜测。

## 4、先可回滚，再重构

重构不是一次性推翻。

更稳妥的方式是：

```shell
保留旧链路
新增新链路
小流量验证
逐步切换
观察指标
确认稳定
删除旧逻辑
```

## 5、先边界清晰，再追求优雅

微服务最重要的不是代码看起来多漂亮，而是职责边界清晰。

边界清楚后，代码、部署、排障、团队协作都会自然变顺。

---

# 总结

微服务架构是一套系统工程，不只是 Spring Cloud、Nacos、Gateway、Sentinel 这些组件的组合。

真正落地时，要同时考虑：

```shell
服务边界
数据归属
调用链路
一致性方案
流量治理
可观测能力
发布回滚
故障补偿
团队协作
```

我的经验是：

```shell
入口要统一
网关要克制
服务要有边界
数据要有归属
消息要可补偿
配置要可治理
故障要可定位
发布要可回滚
```

微服务最终追求的不是“拆得多”，而是让系统在业务持续增长时，仍然能保持清晰、稳定、可演进。
